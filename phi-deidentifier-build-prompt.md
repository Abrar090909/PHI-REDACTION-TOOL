# Project Build Prompt: Clinical Note De-Identification Tool (PHI Redaction)

## Project Title
**MedRedact** — Automated PHI De-identification for Clinical Notes using NER

## One-line Pitch
A tool that takes raw, unstructured clinical notes and automatically detects and redacts Protected Health Information (PHI) — names, dates, locations, ID numbers — making the notes safe for research, analytics, or downstream NLP use, in compliance with HIPAA Safe Harbor guidelines.

---

## Tech Stack

**Backend / ML**
- Python 3.10+
- spaCy (`en_core_web_trf` or `en_core_web_lg`) as base NER engine
- Optionally: `scispacy` or a HuggingFace clinical NER model (`d4data/biomedical-ner-all` or `obi/deid_roberta_i2b2`) for healthcare-specific entities
- FastAPI to serve the model as a REST API (`/redact` endpoint for text, `/redact-image` endpoint for images)
- Regex fallback rules for structured PHI: dates, phone numbers, MRNs, SSNs, ages 90+
- **OCR layer for image input:** `pytesseract` (Tesseract OCR) as the fast/simple option, or `easyocr` if handwriting/scan quality is poor and you need better accuracy. Image → OCR extracts raw text → same NER + regex redaction pipeline runs on that extracted text.

**Frontend**
- React (Vite) + Tailwind CSS
- Single-page app with two input modes:
  - **Text mode:** paste/upload a `.txt` clinical note
  - **Image mode:** upload a photo/scan of a clinical note (`.jpg`/`.png`/`.pdf` page) — shown as a drag-and-drop uploader with an image preview
- Click "Redact" → view side-by-side original vs redacted output (for image mode, show the OCR-extracted raw text alongside the redacted version, so the reviewer can see what OCR picked up before redaction ran)
- Highlight each redacted entity type with a distinct color-coded tag (e.g. `[NAME]` in red, `[DATE]` in blue, `[LOCATION]` in green)
- Simple entity-count summary panel (e.g. "3 names, 2 dates, 1 MRN redacted")
- A small "OCR confidence" note or warning banner if extracted text looks low-quality (e.g. very short output relative to image size) — signals to the user that OCR may have missed something

**Dataset**
- i2b2 2014 De-identification dataset (if accessible via institutional request), OR
- Synthetic clinical notes generated by prompting an LLM to write realistic-sounding fake discharge summaries with labeled PHI (fastest option if i2b2 access is delayed)

## Core Features (MVP scope — keep it tight)

1. **Text input** — paste raw clinical note into a textarea, or upload a `.txt` file
2. **Image input** — upload a photo/scan of a clinical note; backend runs OCR (Tesseract/EasyOCR) to extract raw text before redaction
3. **Redaction engine** — backend runs NER + regex rules, returns entities with character offsets and labels
4. **Side-by-side view** — left panel shows extracted/original text with entities highlighted, right panel shows fully redacted version with placeholder tags
5. **Entity type legend** — color key so a reviewer can see what was caught (NAME, DATE, LOCATION, ID, AGE, PHONE, HOSPITAL)
6. **Download redacted note** — export as `.txt`
7. **Evaluation notebook** (separate from the app) — precision/recall/F1 per entity type against a labeled test set, so you have real numbers to quote in the interview. For image input, additionally track OCR accuracy (word error rate) separately from NER accuracy — this lets you explain the two failure sources independently

---

## What NOT to build (avoid scope creep)
- No user auth / login system
- No database — process is stateless, one note at a time
- No multi-file batch processing (mention it as "future work" if asked)
- No fine-tuning a model from scratch — use pretrained + light fine-tuning only if time allows
- No custom-trained OCR model — use Tesseract/EasyOCR off-the-shelf; don't try to train a handwriting-recognition model, that's a project on its own
- No handwriting-specific tuning — if scanned notes are typed/printed, OCR will work well; if genuinely handwritten, just be upfront in the interview that accuracy will be lower and that's an expected limitation, not a bug

## Interview Talking Points to Prepare
- Why de-identification matters: HIPAA Safe Harbor requires removal of 18 identifier categories before data can be shared or reused
- Why NER over pure regex: regex catches structured PHI (dates, phone numbers) reliably but misses free-text names/locations; NER generalizes better on unstructured clinical language
- Precision vs recall tradeoff: in de-identification, **recall matters more than precision** — missing a real name is a compliance risk, over-redacting a common word is just an inconvenience
- Known challenge: ambiguous entities (e.g., a surname that's also a common English word), and this is where BioBERT/ClinicalBERT-based models outperform general-purpose NER
- Where this fits in the pipeline: it's the enabling layer — necessary before any other clinical NLP tool (summarization, extraction, dictation) can be legally trained on real hospital data
- Why OCR + NER as two separate stages, not one: OCR errors and NER errors are different failure modes. If OCR misreads a word, no NER model can fix that — so you evaluate and report on them separately (OCR word error rate vs NER precision/recall) rather than treating the whole pipeline as one black box
- Realistic limitation to state upfront: OCR on printed/typed scanned notes works well; genuinely messy handwriting will lower accuracy meaningfully — you'd flag this as a known constraint rather than pretend it's solved

## Build Order (fastest path to a demo-able app)
1. Get NER working in a plain Python script on 2-3 sample notes — confirm entity detection works
2. Wrap it in a FastAPI endpoint (`/redact`)
3. Add OCR: test `pytesseract` on a couple of sample note images/screenshots, confirm extracted text is usable, wrap in `/redact-image` endpoint that pipes OCR output into the same redactor
4. Build the React UI with both input modes (text paste + image upload) hitting the respective endpoints, rendering highlighted output
5. Run the evaluation notebook to get real F1 numbers (NER) and word error rate (OCR)
6. Polish UI (legend, download button, entity counts, OCR confidence banner)
7. Write a short README with the pitch, architecture diagram, and sample screenshots
